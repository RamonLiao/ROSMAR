# ROSMAR CRM — P4 Design Document

**Date**: 2026-03-07
**Status**: Approved
**Baseline**: P3 complete (35 tasks, 136 BFF tests, 111 frontend tests)

---

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Escrow complexity | Full suite (multi-token concept, partial release, milestone vesting, arbitration) | Real token deal use case demands it |
| Escrow token support | `Balance<SUI>` only in v1; multi-token `Escrow<phantom T>` deferred to P5 | Matches codebase pattern (reward.move), avoids generic complexity |
| Quest verification | Hybrid (indexer + RPC + manual) + SBT badge | Pragmatic given indexer is stub; SBT is composable proof |
| Quest badge revocation | No revocation in v1 (SBT = permanent) | SUI owned object model makes admin revocation impractical without holder cooperation |
| Lookalike algorithm | K-NN cosine similarity, internal candidates only | Better than rules, doesn't need graph data; leave interface for future C + on-chain discovery |
| GDPR approach | PII nullify + Seal key destroy + 30-day grace period + audit trail | Industry standard for Web3 CRM; satisfies EDPB guidance on blockchain immutability |
| Production hardening | Cache + rate limit + observability (Sentry + pino) | Minimum viable production; security audit prep deferred to P5 |
| Wave strategy | Move + Prisma foundation first, BFF/frontend parallel after | Proven in P3 Wave 1; eliminates schema merge conflicts |

### P5 Backlog (recorded for future)

- Multi-token escrow: `Escrow<phantom T>` with `Balance<T>`
- Lookalike: Graph-based similarity (C) + on-chain wallet discovery
- Security audit prep: SlowMist/CertiK checklist, OWASP top 10 scan, penetration test
- Quest badge revocation: Receiving pattern or shared object registry

### VestingSchedule (dynamic field on Escrow)

```move
public struct VestingSchedule has store {
    vesting_type: u8,           // LINEAR=0, MILESTONE=1
    cliff_ms: u64,
    total_duration_ms: u64,     // for LINEAR

    description: String,

    completed_at: u64,
```

Linear vesting calculation uses `u128` intermediate to prevent overflow:
```move
fun calc_linear_vested(total: u64, elapsed_ms: u64, duration_ms: u64): u64 {
    if (elapsed_ms >= duration_ms) return total;
    let vested = (total as u128) * (elapsed_ms as u128) / (duration_ms as u128);
    (vested as u64)
}
```

### ArbitrationState (dynamic field, created on raise_dispute)

```move
public struct ArbitrationState has store {
    votes_release: vector<address>,
    votes_refund: vector<address>,
    resolved: bool,
    resolution: Option<u8>,     // 0=RELEASE, 1=REFUND
}
```

Dispute uses on-chain per-arbitrator voting (not off-chain signatures):
- `vote_on_dispute(escrow, decision, ctx)` — arbitrator calls, vote recorded
- When votes reach threshold -> auto-execute release or refund

### Entry Functions

| Function | Caller | From State | To State |
|----------|--------|------------|----------|
| `create_escrow` | WorkspaceAdminCap | - | CREATED |
| `fund_escrow(coin)` | payer | CREATED | FUNDED |
| `add_vesting(schedule)` | payer | FUNDED | FUNDED (vesting attached) |
| `release(amount)` | payer | FUNDED/PARTIALLY_RELEASED | PARTIALLY_RELEASED/COMPLETED |
| `release_vested(clock)` | payer | FUNDED/PARTIALLY_RELEASED | PARTIALLY_RELEASED/COMPLETED |
| `complete_milestone(idx)` | payer | FUNDED/PARTIALLY_RELEASED | same |
| `refund()` | payer | CREATED/FUNDED (pre-dispute, or expired) | REFUNDED |
| `raise_dispute()` | payer or payee | FUNDED/PARTIALLY_RELEASED | DISPUTED |
| `vote_on_dispute(decision)` | arbitrator | DISPUTED | DISPUTED/COMPLETED/REFUNDED |

Guards:
- All functions: `assert_not_paused(config)`, `assert_state()`, caller auth
- `fund_escrow`: MIN_LOCK_DURATION for flash loan prevention
- `arbitrators` immutable after FUNDED
- `release`: `amount <= balance.value()`, respects vesting if attached
- `vote_on_dispute`: dedup votes, auto-execute when threshold met
- Escrow is a **shared object** (arbitrators need mutation access)

### Events

`EscrowCreated`, `EscrowFunded`, `EscrowReleased`, `EscrowRefunded`, `MilestoneCompleted`, `DisputeRaised`, `DisputeVoteCast`, `DisputeResolved` — all include `workspace_id`, `escrow_id`, `actor`, `timestamp` (consistent with `AuditEventV1` pattern).

### BFF: Deal Stage State Machine

```
LEAD -> QUALIFIED -> PROPOSAL -> NEGOTIATION -> WON -> CLOSED
                                             -> LOST -> CLOSED
                                      <-> DISPUTED (when escrow disputed)
```

Escrow-deal linkage:
- `NEGOTIATION` -> can create escrow
- `WON` -> trigger release or release_vested
- `LOST` -> trigger refund
- Escrow `DISPUTED` -> deal marked `DISPUTED`

## Section 3: Lookalike Audiences

### Algorithm: K-NN Cosine Similarity

Feature vector per profile (6 dimensions):

| Dimension | Source | Normalization |
|-----------|--------|---------------|
| engagement_score | Profile.engagementScore | min-max [0,1] |
| tx_frequency | wallet_events count / 30d | log scale [0,1] |
| asset_diversity | unique token types held | min-max [0,1] |
| nft_count | NFT holdings count | log scale [0,1] |
| defi_activity | swap+stake+LP events / 30d | log scale [0,1] |
| dao_participation | vote events / 30d | min-max [0,1] |

Flow:
1. Admin selects seed segment
2. Compute centroid (average feature vector of seed)
3. For all other workspace profiles: compute cosine similarity to centroid
4. Rank by similarity, return top K (default 50)
5. Optionally auto-create segment from results

### Extensibility Interface (for future Graph + On-chain Discovery)

```typescript
interface SimilarityStrategy {
  name: string;  // 'knn-cosine' | 'graph-based'
  findSimilar(
    seedProfiles: ProfileFeatureVector[],
    candidatePool: ProfileFeatureVector[],

interface CandidateSource {
  name: string;  // 'internal' | 'on-chain-discovery'

```

v1: `KnnCosineSimilarity` + `InternalCandidateSource`
Future: plug in `GraphSimilarity` + `OnChainCandidateSource` without changing existing code.

## Section 4: GDPR Compliance

### Deletion Flow (30-day grace period)

```
DELETE /profiles/:id/gdpr
  -> gdprStatus = PENDING_DELETION, scheduledAt = now + 30 days
  -> Profile hidden from UI immediately

After 30 days (GdprCleanupJob cron):
  1. PII nullification: name, email, notes -> null
  2. Social links: delete all SocialLink rows
  3. Wallet unbinding: ProfileWallet.profileId -> null
  4. Seal key destruction: VaultSecrets -> destroy Seal policy keys
  5. Deal documents: destroy Seal keys (docs unreadable)
  6. Agent logs: purge AI usage logs containing profile data
  7. Quest completions: anonymize (profileId -> null, keep aggregate stats)
  8. Segment memberships: remove from all segments
  9. Audit trail: write GdprDeletionLog (immutable, no PII)
  10. Profile remains as tombstone: { id, workspaceId, isDeleted: true, deletedAt }
```

## Section 5: Production Hardening

| Key Pattern | Data | TTL | Invalidation |
|-------------|------|-----|-------------|
| `profile:{id}` | Profile + wallets + social | 5min | On update/GDPR delete |
